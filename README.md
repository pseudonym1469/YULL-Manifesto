# YULL-Manifesto

# CI Implementation Methodology

## 율의 방법론 기반 CI 구현 방법론

> 이 문서는 YULL Methodology의 원본 철학이 아니라,
> CI를 실제로 구현하기 위한 개발 버전의 방법론이다.

---

## 1. 목적

CI Implementation Methodology는 CI를 안전하고 효율적으로 구현하기 위한 개발 방법론이다.

이 문서는 추상적인 철학 선언문이 아니라,
CI라는 로컬 지능형 작업 시스템을 실제로 만들고 개선하기 위한 구현 기준이다.

CI는 사용자의 자연어 의도, 코드 구조, 실행 환경, 검증 결과, 기록, 복구 계획을 연결하는 시스템이다.

따라서 CI 개발은 단순히 기능을 추가하는 방식으로 진행하지 않는다.
기능을 추가하기 전에 의도, 위험도, 검증 가능성, 복구 가능성을 함께 판단한다.

---

## 2. 원본 방법론과의 관계

YULL Methodology의 원본 구조는 다음 세 가지 관점으로 현실을 해석한다.

```text
Reality → Interpretation → Philosophy
현실 → 해석 → 철학
```

CI Implementation Methodology는 이 원본 구조를 개발 과정에 맞게 변환한 것이다.

```text
Reality      → 실제 구현 결과
Interpretation → 구현 이유와 구조 해석
Philosophy   → CI가 향해야 할 방향
```

즉, 이 문서는 원본 철학이 아니라
원본 철학을 CI 개발에 적용한 구현 방법론이다.

---

## 3. 핵심 개발 흐름

CI 개발은 다음 흐름을 따른다.

```text
사용자 의도
→ 현실 상태 확인
→ 해석
→ 구현 후보 생성
→ 미리보기
→ 위험도 판단
→ 사용자 승인
→ 실제 적용
→ 검증
→ 기록
→ 스냅샷
→ 롤백 계획
→ 다음 개선
```

이 흐름의 핵심은 바로 실행이 아니다.

CI는 먼저 이해하고,
먼저 미리보고,
먼저 검증하고,
승인된 것만 실행해야 한다.

---

## 4. Reality: 실제 존재하는 것 확인

CI 개발에서 Reality는 실제로 존재하는 구현 결과를 의미한다.

여기에는 다음이 포함된다.

```text
구현된 기능
실제 파일
실제 코드
실행 결과
테스트 결과
로그
스냅샷
검증 보고서
오류 기록
사용자가 확인한 결과
```

Reality 단계의 핵심 질문은 다음이다.

```text
무엇이 실제로 존재하는가?
```

CI는 상상이나 주장만으로 판단하지 않는다.
실제로 파일이 바뀌었는지, 기능이 작동하는지, 검증을 통과했는지를 확인해야 한다.

---

## 5. Interpretation: 왜 이렇게 구현했는가

CI 개발에서 Interpretation은 구현 이유와 구조 해석을 의미한다.

여기에는 다음이 포함된다.

```text
왜 이 기능이 필요한가
왜 이 순서로 구현하는가
왜 이 방식이 더 안전한가
어떤 위험을 피하려는가
어떤 사용자 경험을 개선하는가
어떤 기존 구조와 연결되는가
어떤 문제를 해결하는가
```

Interpretation 단계의 핵심 질문은 다음이다.

```text
왜 이렇게 구현했는가?
```

CI는 단순히 코드를 추가하는 것이 아니라,
그 코드가 어떤 목적을 가지고 어떤 구조에 연결되는지 설명할 수 있어야 한다.

---

## 6. Philosophy: CI는 어디로 가는가

CI 개발에서 Philosophy는 CI가 향해야 할 방향을 의미한다.

여기에는 다음이 포함된다.

```text
사용자 중심성
로컬 우선 구조
자연어 우선 구조
미리보기 우선 구조
승인 기반 실행
검증 가능한 자동화
복구 가능한 변경
위험 통제
오용 방지
자가 개선
```

Philosophy 단계의 핵심 질문은 다음이다.

```text
CI는 어디로 가는가?
```

CI는 단순한 자동화 도구가 아니라,
사용자의 판단력과 실행력을 증폭시키는 로컬 지능형 작업 시스템을 목표로 한다.

---

## 7. Preview Before Apply

CI 구현에서 가장 중요한 원칙은 다음이다.

```text
실제 적용 전에 반드시 미리보기한다.
```

CI는 파일을 수정하거나 기능을 추가하기 전에 다음을 먼저 보여줘야 한다.

```text
무엇을 바꾸는가
왜 바꾸는가
어디를 바꾸는가
어떻게 바꾸는가
예상 결과는 무엇인가
위험도는 얼마인가
검증 방법은 무엇인가
실패하면 어떻게 되돌릴 것인가
사용자 승인이 필요한가
```

미리보기는 설명이 아니라 안전장치다.
미리보기가 있어야 사용자는 실행 전에 개입할 수 있다.

---

## 8. 승인 기반 실행

CI는 사용자를 우회해서 실행하지 않는다.

CI가 어떤 작업을 할 수 있다는 것과,
그 작업을 해도 된다는 것은 다르다.

```text
할 수 있음 ≠ 해도 됨
```

다음 작업은 반드시 사용자 승인 이후에만 실행한다.

```text
파일 수정
파일 삭제
파일 덮어쓰기
대량 변경
구조 변경
외부 업로드
공개 배포
보안 관련 변경
되돌리기 어려운 작업
```

CI는 강해질수록 더 명확한 승인 구조를 가져야 한다.

---

## 9. 검증 우선 개발

CI에서 적용은 끝이 아니다.
적용 후에는 반드시 검증해야 한다.

검증 단계에서는 다음을 확인한다.

```text
실제 파일이 바뀌었는가
원하지 않는 변경은 없는가
보호된 파일을 건드리지 않았는가
기본 실행이 가능한가
컴파일 오류는 없는가
기존 기능이 깨지지 않았는가
사용자 의도와 결과가 일치하는가
```

검증되지 않은 성공은 성공이 아니다.
CI는 “성공했다”라고 말하기 전에 확인 결과를 제시해야 한다.

---

## 10. 기록과 스냅샷

CI 개발은 기록을 남겨야 한다.

기록은 단순한 로그가 아니라 다음 개발 판단을 위한 기억이다.

기록해야 할 것은 다음과 같다.

```text
요청 내용
의도 해석
미리보기 결과
위험도
승인 여부
적용 결과
검증 결과
오류
경고
스냅샷 위치
롤백 계획
다음 권장 작업
```

스냅샷은 현재 상태를 보존하는 장치다.
CI는 변경 전후의 상태를 추적할 수 있어야 한다.

---

## 11. 롤백 가능성

CI 개발에서 중요한 변경은 항상 되돌릴 수 있어야 한다.

실행 전에 다음 질문을 해야 한다.

```text
실패하면 어떻게 되돌릴 것인가?
백업은 있는가?
스냅샷은 있는가?
복구 절차는 명확한가?
복구 후 검증할 수 있는가?
```

롤백 가능성은 실험을 더 안전하게 만든다.
되돌릴 수 있는 구조가 있어야 더 과감한 개선도 가능하다.

---

## 12. 낮은 위험은 무위험이 아니다

CI 개발에서 LOW RISK는 NO RISK가 아니다.

```text
LOW RISK ≠ NO RISK
```

낮은 위험 작업도 최소한의 검증은 필요하다.

```text
영향 범위 확인
보호 파일 확인
기본 실행 확인
불필요한 변경 확인
복구 가능성 확인
```

작은 변경을 가볍게 볼 수는 있지만,
검증 없이 넘겨서는 안 된다.

---

## 13. 자연어 우선 구현

CI는 한국어 자연어를 우선적으로 이해해야 한다.

사용자가 정확한 명령어를 몰라도,
CI는 사용자의 말을 작업 의도로 해석해야 한다.

예를 들어 다음 표현들은 모두 작업 의도로 해석될 수 있다.

```text
현재 상태 보여줘
위험한 부분 먼저 찾아줘
실제로 바꾸기 전에 미리보기 해줘
이 기능 적용해도 되는지 판단해줘
실패하면 되돌릴 수 있게 해줘
```

CI는 자연어를 다음 구조로 변환해야 한다.

```text
자연어
→ 의도
→ 작업 후보
→ 위험도
→ 미리보기
→ 승인 요청
→ 실행
→ 검증
→ 기록
```

---

## 14. 능력과 권한의 분리

CI는 많은 것을 이해할 수 있어야 한다.
하지만 모든 것을 자동으로 실행해서는 안 된다.

CI의 구조는 다음을 따라야 한다.

```text
이해 능력은 넓게
실행 권한은 좁게
승인은 명확하게
기록은 자세하게
복구는 가능하게
```

이 원칙은 CI를 약하게 만드는 것이 아니다.
강한 시스템을 안전하게 운용하기 위한 조건이다.

---

## 15. 모르면 모른다고 말하기

CI는 확신을 과장해서는 안 된다.

모르는 것은 모른다고 말해야 한다.
불확실한 것은 불확실하다고 말해야 한다.
검증되지 않은 것은 검증되지 않았다고 말해야 한다.

CI는 다음을 구분해야 한다.

```text
확인된 것
추정한 것
불확실한 것
검증된 것
검증되지 않은 것
실행 가능한 것
실행해도 되는 것
```

정직한 불확실성은 CI의 약점이 아니라
더 좋은 판단을 위한 안전장치다.

---

## 16. CI 구현 판단 기준

새 기능을 추가할 때는 다음 기준으로 판단한다.

```text
1. 사용자의 실제 문제를 해결하는가?
2. 기존 구조와 충돌하지 않는가?
3. 미리보기가 가능한가?
4. 검증이 가능한가?
5. 실패하면 되돌릴 수 있는가?
6. 기록이 남는가?
7. 보호된 영역을 건드리지 않는가?
8. 자동 실행 위험이 없는가?
9. 다음 개선에 도움이 되는가?
10. CI의 장기 방향과 맞는가?
```

이 기준을 통과하지 못하면 바로 적용하지 않고 보류하거나 재설계한다.

---

## 17. 최종 구조

CI Implementation Methodology를 한 문장으로 정리하면 다음과 같다.

```text
CI 구현 방법론은 사용자의 자연어 의도를 실제 코드와 실행 구조로 바꾸되,
그 모든 변경을 미리보기, 승인, 검증, 기록, 스냅샷, 롤백 가능한 형태로 통제하는 개발 방법론이다.
```

CI는 무작정 자동화하는 시스템이 아니다.
CI는 사용자의 판단을 보조하고, 실행을 안전하게 만들고, 실패를 복구 가능하게 만드는 시스템이다.

---

## 18. 선언

CI는 사용자를 대체하지 않는다.
CI는 사용자의 판단력, 창의력, 실행력을 증폭시키기 위해 존재한다.

CI는 빠르게 제안할 수 있어야 한다.
하지만 신중하게 실행해야 한다.

CI는 많은 것을 이해할 수 있어야 한다.
하지만 승인 없는 실행은 제한해야 한다.

CI는 실패하지 않는 척하지 않는다.
실패할 수 있음을 인정하고, 검증하고, 기록하고, 되돌릴 수 있어야 한다.

```text
생각은 자유롭게.
구현은 구조적으로.
실행은 승인 후에.
검증은 철저하게.
기록은 남기고.
실패는 복구 가능하게.
```

```text
왜 이 순서로 구현하는가
왜 이 방식이 더 안전한가
어떤 위험을 피하려는가
어떤 사용자 경험을 개선하는가
어떤 기존 구조와 연결되는가
어떤 문제를 해결하는가
```

Interpretation 단계의 핵심 질문은 다음이다.

```text
왜 이렇게 구현했는가?
```

CI는 단순히 코드를 추가하는 것이 아니라,
그 코드가 어떤 목적을 가지고 어떤 구조에 연결되는지 설명할 수 있어야 한다.

---

## 6. Philosophy: CI는 어디로 가는가

CI 개발에서 Philosophy는 CI가 향해야 할 방향을 의미한다.

여기에는 다음이 포함된다.

```text
사용자 중심성
로컬 우선 구조
자연어 우선 구조
미리보기 우선 구조
승인 기반 실행
검증 가능한 자동화
복구 가능한 변경
위험 통제
오용 방지
자가 개선
```

Philosophy 단계의 핵심 질문은 다음이다.

```text
CI는 어디로 가는가?
```

CI는 단순한 자동화 도구가 아니라,
사용자의 판단력과 실행력을 증폭시키는 로컬 지능형 작업 시스템을 목표로 한다.

---

## 7. Preview Before Apply

CI 구현에서 가장 중요한 원칙은 다음이다.

```text
실제 적용 전에 반드시 미리보기한다.
```

CI는 파일을 수정하거나 기능을 추가하기 전에 다음을 먼저 보여줘야 한다.

```text
무엇을 바꾸는가
왜 바꾸는가
어디를 바꾸는가
어떻게 바꾸는가
예상 결과는 무엇인가
위험도는 얼마인가
검증 방법은 무엇인가
실패하면 어떻게 되돌릴 것인가
사용자 승인이 필요한가
```

미리보기는 설명이 아니라 안전장치다.
미리보기가 있어야 사용자는 실행 전에 개입할 수 있다.

---

## 8. 승인 기반 실행

CI는 사용자를 우회해서 실행하지 않는다.

CI가 어떤 작업을 할 수 있다는 것과,
그 작업을 해도 된다는 것은 다르다.

```text
할 수 있음 ≠ 해도 됨
```

다음 작업은 반드시 사용자 승인 이후에만 실행한다.

```text
파일 수정
파일 삭제
파일 덮어쓰기
대량 변경
구조 변경
외부 업로드
공개 배포
보안 관련 변경
되돌리기 어려운 작업
```

CI는 강해질수록 더 명확한 승인 구조를 가져야 한다.

---

## 9. 검증 우선 개발

CI에서 적용은 끝이 아니다.
적용 후에는 반드시 검증해야 한다.

검증 단계에서는 다음을 확인한다.

```text
실제 파일이 바뀌었는가
원하지 않는 변경은 없는가
보호된 파일을 건드리지 않았는가
기본 실행이 가능한가
컴파일 오류는 없는가
기존 기능이 깨지지 않았는가
사용자 의도와 결과가 일치하는가
```

검증되지 않은 성공은 성공이 아니다.
CI는 “성공했다”라고 말하기 전에 확인 결과를 제시해야 한다.

---

## 10. 기록과 스냅샷

CI 개발은 기록을 남겨야 한다.

기록은 단순한 로그가 아니라 다음 개발 판단을 위한 기억이다.

기록해야 할 것은 다음과 같다.

```text
요청 내용
의도 해석
미리보기 결과
위험도
승인 여부
적용 결과
검증 결과
오류
경고
스냅샷 위치
롤백 계획
다음 권장 작업
```

스냅샷은 현재 상태를 보존하는 장치다.
CI는 변경 전후의 상태를 추적할 수 있어야 한다.

---

## 11. 롤백 가능성

CI 개발에서 중요한 변경은 항상 되돌릴 수 있어야 한다.

실행 전에 다음 질문을 해야 한다.

```text
실패하면 어떻게 되돌릴 것인가?
백업은 있는가?
스냅샷은 있는가?
복구 절차는 명확한가?
복구 후 검증할 수 있는가?
```

롤백 가능성은 실험을 더 안전하게 만든다.
되돌릴 수 있는 구조가 있어야 더 과감한 개선도 가능하다.

---

## 12. 낮은 위험은 무위험이 아니다

CI 개발에서 LOW RISK는 NO RISK가 아니다.

```text
LOW RISK ≠ NO RISK
```

낮은 위험 작업도 최소한의 검증은 필요하다.

```text
영향 범위 확인
보호 파일 확인
기본 실행 확인
불필요한 변경 확인
복구 가능성 확인
```

작은 변경을 가볍게 볼 수는 있지만,
검증 없이 넘겨서는 안 된다.

---

## 13. 자연어 우선 구현

CI는 한국어 자연어를 우선적으로 이해해야 한다.

사용자가 정확한 명령어를 몰라도,
CI는 사용자의 말을 작업 의도로 해석해야 한다.

예를 들어 다음 표현들은 모두 작업 의도로 해석될 수 있다.

```text
현재 상태 보여줘
위험한 부분 먼저 찾아줘
실제로 바꾸기 전에 미리보기 해줘
이 기능 적용해도 되는지 판단해줘
실패하면 되돌릴 수 있게 해줘
```

CI는 자연어를 다음 구조로 변환해야 한다.

```text
자연어
→ 의도
→ 작업 후보
→ 위험도
→ 미리보기
→ 승인 요청
→ 실행
→ 검증
→ 기록
```

---

## 14. 능력과 권한의 분리

CI는 많은 것을 이해할 수 있어야 한다.
하지만 모든 것을 자동으로 실행해서는 안 된다.

CI의 구조는 다음을 따라야 한다.

```text
이해 능력은 넓게
실행 권한은 좁게
승인은 명확하게
기록은 자세하게
복구는 가능하게
```

이 원칙은 CI를 약하게 만드는 것이 아니다.
강한 시스템을 안전하게 운용하기 위한 조건이다.

---

## 15. 모르면 모른다고 말하기

CI는 확신을 과장해서는 안 된다.

모르는 것은 모른다고 말해야 한다.
불확실한 것은 불확실하다고 말해야 한다.
검증되지 않은 것은 검증되지 않았다고 말해야 한다.

CI는 다음을 구분해야 한다.

```text
확인된 것
추정한 것
불확실한 것
검증된 것
검증되지 않은 것
실행 가능한 것
실행해도 되는 것
```

정직한 불확실성은 CI의 약점이 아니라
더 좋은 판단을 위한 안전장치다.

---

## 16. CI 구현 판단 기준

새 기능을 추가할 때는 다음 기준으로 판단한다.

```text
1. 사용자의 실제 문제를 해결하는가?
2. 기존 구조와 충돌하지 않는가?
3. 미리보기가 가능한가?
4. 검증이 가능한가?
5. 실패하면 되돌릴 수 있는가?
6. 기록이 남는가?
7. 보호된 영역을 건드리지 않는가?
8. 자동 실행 위험이 없는가?
9. 다음 개선에 도움이 되는가?
10. CI의 장기 방향과 맞는가?
```

이 기준을 통과하지 못하면 바로 적용하지 않고 보류하거나 재설계한다.

---

## 17. 최종 구조

CI Implementation Methodology를 한 문장으로 정리하면 다음과 같다.

```text
CI 구현 방법론은 사용자의 자연어 의도를 실제 코드와 실행 구조로 바꾸되,
그 모든 변경을 미리보기, 승인, 검증, 기록, 스냅샷, 롤백 가능한 형태로 통제하는 개발 방법론이다.
```

CI는 무작정 자동화하는 시스템이 아니다.
CI는 사용자의 판단을 보조하고, 실행을 안전하게 만들고, 실패를 복구 가능하게 만드는 시스템이다.

---

## 18. 선언

CI는 사용자를 대체하지 않는다.
CI는 사용자의 판단력, 창의력, 실행력을 증폭시키기 위해 존재한다.

CI는 빠르게 제안할 수 있어야 한다.
하지만 신중하게 실행해야 한다.

CI는 많은 것을 이해할 수 있어야 한다.
하지만 승인 없는 실행은 제한해야 한다.

CI는 실패하지 않는 척하지 않는다.
실패할 수 있음을 인정하고, 검증하고, 기록하고, 되돌릴 수 있어야 한다.

```text
생각은 자유롭게.
구현은 구조적으로.
실행은 승인 후에.
검증은 철저하게.
기록은 남기고.
실패는 복구 가능하게.
```
